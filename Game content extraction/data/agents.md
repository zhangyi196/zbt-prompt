# Data Agents Guide

本目录不是流程脚本集合，而是数据层规则入口。
它服务 `blind_boxes.py`、`animals.py`、`item_states.py` 等事实源维护，负责给 AI 一个稳定的总判断框架，而不是逐步操作表。

当条目虽然过了测试，但仍然不自然、模板化、伪多样或不适合找不同时，不得停在“规则允许”；必须继续复查并继续修改。

## 1. 文档定位

- 本文件是数据层总入口，不重复维护专项规则里的阈值、示例和家族清单。
- 它规定目标、边界、优先级、必读顺序和统一验收口径。
- 具体怎么发现问题，看 `审查规则.md`。
- 具体怎么替换落库，看 `修改规则.md`。
- 审查文档命名与固定结构，看 `审查文件.md`。
- 一轮修改完成后的通用收尾复查，看 `复查清单.md`。
- 跨类别高频重复词的统一定位与整改索引，看 `全局重复词定位清单.md`。
- 跨类别不合理命名的统一定位与整改索引，看 `全局不合理命名清单.md`。

## 2. 文件职责

- `blind_boxes.py`：20 个盲盒 `场景+用途` 入口、四池物品库、兼容映射和风险排除配置。
- `审查规则.md`：审查任务宪法，负责界定什么应判为问题、什么不能放行。
- `审查文件.md`：审查文档索引、命名规则和固定输出结构。
- `审查文件/{类别} 审查文档.md`：该类别的问题记录、替换方向、修改复审和剩余风险。
- `修改规则.md`：修改任务宪法，负责界定替换目标、边界和验收标准。
- `复查清单.md`：修改完成后的精简收尾检查表，负责统一确认“还能不能收尾”。
- `全局重复词定位清单.md`：跨类别高频重复词、词族和定位记录；用于后续统一整改，不替代单类别审查文档。
- `全局不合理命名清单.md`：跨类别不直观、功能先于主体、壳体先于本体或解释成本过高的命名记录；用于后续统一整改，不替代单类别审查文档。
- `animals.py`：动物类型、动物用品和动物痕迹数据。
- `item_states.py`：物品状态词；透明、反光、高光语义视为风险。
- `__init__.py`：数据包标记文件。

## 3. 工作顺序

修改盲盒内容时，默认顺序是：

1. 先读本文件，确认总目标、优先级和禁区。
2. 再读 `审查规则.md`，确认问题模式、否决条件和审查输出要求。
3. 再读 `审查文件.md` 与对应类别审查文档，确认该类现状、必换项和历史判断。
4. 再读 `修改规则.md`，确认替换边界和通过标准。
5. 修改 `blind_boxes.py`，并回写对应类别审查文档。
6. 读 `复查清单.md` 做收尾复查，再补写类别审查文档的复审结论。
7. 若发现跨类别反复出现的不直观命名或解释型命名，同步记入 `全局不合理命名清单.md`。

## 4. 数据层总目标

本目录服务找不同游戏，不服务场景装饰、百科罗列或关键词凑数。

所有数据修改都应朝下面目标收敛：

- 物品真实可见。
- 场景自然稳定。
- 四池角色清楚。
- 差异点单一明确。
- 命名先让人看到主体物，普通玩家第一眼能成像。
- 多样性来自物体本体差异，不来自换前缀、换内容词或换承载壳。
- 若主词未变，只是在前面补场景词、口味词或材质词，如 `磅蛋糕 -> 柠檬磅蛋糕`、`茶包 -> 篝火茶包`，默认仍按同物处理。
- 若主体物单独写不够直观，只能在内容本身清楚可见时补成“可见内容 + 主体物”；若读起来像说明句，仍视为失败。

## 5. 优先级总纲

当多个判断冲突时，按下面顺序裁决，低优先级不得推翻高优先级：

1. `真实与自然`
   数据首先应像真实场景物，而不是测试友好词。
2. `场景语境`
   条目必须属于当前类别前台，而不是后场、维护、仓储、安装、清洁、冷却、晾放或专业流程。
3. `物体本体`
   条目必须先是独立成物的对象，而不是内容词、功能词或壳体词。
4. `四池角色`
   条目必须符合所在池的职责，尤其第四池必须是“场景补充中型物”。
5. `可玩性`
   条目必须适合做找不同中的单点差异。
6. `多样性`
   多样性必须来自真实物族和视觉本体差异，而不是模板差异。
7. `数量与测试`
   数量、唯一性、兼容映射和自动测试必须通过，但它们只是底线，不是生成方向。

一句话原则：
`像一眼能看见的真实物，优先于像规则答案。`

## 6. 四池总契约

`BLIND_BOX_ITEM_POOL_BUNDLES` 的每个类别固定包含四个列表：

```python
{
    "core_items": [],
    "support_items": [],
    "visible_small_items": [],
    "scene_expansion_items": [],
}
```

数据层必须长期保持：

- 每池 50 条。
- 池内唯一。
- 字段名不变。
- 与运行时兼容映射一致。

兼容关系保持不变：

- `core_items` 和 `scene_expansion_items` 映射到大型物品。
- `support_items` 加前 6 个 `core_items` 映射到中型物品。
- `visible_small_items` 映射到散落小型物品。
- `hanging` 为空兼容桶，不应反向指导写库。

总定义：

- `core_items` 写主体物、主承载物、主容器、主设备。
- `support_items` 写服务核心物的常见工具、辅助件、取用件、承托件、替换件。
- `visible_small_items` 写升级成清晰可见单位的小物集合，如一盒、一包、一筒、一盘、一套、一卷、一束、一叠、一册、一排。
- `scene_expansion_items` 写普通玩家一眼能认出的前台自然中型实物，优先单个实物；全池只允许极少量简单组合，最多 5 条，且每条组合最多 3 个内容，仍要一眼读成一个整体。

## 7. 全局硬禁区

以下内容默认不应进入盲盒物品池：

- 抽象词、概括词、纯用途词、纯区域词、纯动作词。
- 依赖文字、标签、价格签、贴纸、细线、小点、刻痕、碎屑、污渍才能识别的内容。
- 依赖光影、发光、高光、反光、透明、投影、镜面成像变化的内容。
- 单个微小物：针、钉、扣、环、珠、粒、芯、帽、签、贴、线头、碎片、小卡片。
- 大型家具、大面积主体或难以自由放入画面的结构物。
- 同一基础对象在同一类别四池机械重复。
- 只靠同质尾词撑起来的凑数条目。
- 主要靠说明前缀撑起来的弱本体条目，如 `xx板 / xx垫 / xx盘 / xx盒 / xx册`。
- 前缀换词、后缀不变的模板复写条目，如 `项目记录本 / 作品记录本 / 行数记录本`。
- 场景词 / 口味词 / 材质词 + 原主词的前缀缝合条目，如 `柠檬磅蛋糕 / 篝火茶包 / 徒步燕麦棒`。

第四池额外硬禁区：

- 复杂组合、散堆组合或超过 3 个内容的组合
- 核心物的近义变体、材质变体、尺寸变体、形状变体
- 二级器具化条目，如抽取器、托架、收纳座、保温座、分区盒、立筒、挂架、临放盘
- 架子类、箱子类、柜子类、支架类的常规扩写条目
- 内容承载壳化条目，如 `内容词 + 夹 / 筒 / 袋 / 盒`
- 为了过中型边界而临时补壳体后缀的条目

## 8. 审查与修改分工

审查和修改是两个阶段，但共享同一套高层判断：

- 统一修改目标：命名必须让普通玩家第一眼成像，而不是先理解用途、流程或分类。
- 审查负责发现根因，不负责用换词掩盖问题。
- 修改负责解决根因，不负责把问题包装成“已通过”。
- 只要结果仍然机械、别扭、壳体化、器具化或伪多样，两个阶段都必须继续复查。
- 同族平移替换不算修复，例如 `作品记录本 -> 项目说明本`、`海流记录板 -> 潜点提示板` 都不算修改成功。
- 如果旧坏家族只是被替换成另一串 `xx板 / xx架 / xx器 / xx机 / xx本 / xx册` 弱模板词，视为假修复，必须继续重做。

## 9. 审查文档契约

- 同一类别只维护一个 `审查文件/{类别} 审查文档.md`，不要另起临时总结替代它。
- 审查阶段至少写清：`问题总览`、`按池问题与必换项`、`多样性与模板密度`、`第四池专项`、`修改方向`。
- 修改阶段必须补写：`本次修改记录`、`修改后复审`、`剩余风险与暂留理由`。
- 审查文档必须让下次会话不回看对话也能直接继续工作。

## 10. 国际化边界

- 默认面向欧美 / 国际化找不同用户。
- 中文只是输出语言，不等于物品本体可以是强地域专属。
- 除非类别明确要求，否则默认避免强中式、强地域、强课程体系绑定物品。
- 品牌名、商标名、IP 名称默认不写。

## 11. 验收口径

数据层修改只有在下面条件同时满足时，才算通过：

- 四池数量正确，池内唯一。
- 条目真实、自然、第一眼可识别，且名字优先呈现主体物。
- 条目适合找不同单点差异。
- 四池角色没有混乱，尤其第四池没有滑向器具池和壳体池。
- 伪多样性被压住，而不是仅仅换了名字。
- 默认符合国际化场景常识。
- 自动测试、兼容结构和必要回归检查通过。

## 12. 最终判语

合格的数据层规则不是“把所有动作写细”，而是：

- 让 AI 知道什么才是目标；
- 让 AI 知道什么一票否决；
- 让 AI 知道冲突时先服从什么；
- 让 AI 在结果有问题时继续复查，而不是停在“已经执行完步骤”。

如果一个结果虽然过了测试，但仍然让人一眼看出是“为了过规则临时造的”，就视为未通过，必须继续审查或继续修改。
