# 盲盒物品库审查规则

本文档用于审查 `blind_boxes.py` 中已写入的盲盒物品内容。审查目标不是只保证格式正确，而是确保物品真实、常见、可识别、适合找不同游戏，并且不会通过换前缀制造低质量重复。

每次执行审查后，必须把结果写入 `审查文件/{类别} 审查文档.md`。该类别文档是后续修改阶段的唯一问题清单，不能只在对话中描述审查结论，也不能把多个类别混写到同一个审查文档。

## 审查原则

- 按“视觉本体”审查，而不是只按名称审查；名称不同但画出来基本是同一种东西，视为实质重复。
- 优先保留欧美/国际化用户常见、容易识别、适合画面摆放的物品。
- 避免过小、过薄、过碎、过度依赖文字或标签才能识别的物品。
- 避免用容器、袋子、托盘、架子、盒子等模板词批量凑数。
- 第四池 `scene_expansion_items` 保留历史字段名，但实际含义是“可放置中型单物”，不得写组合。

## 1. 内容合理性

逐项检查物品是否合理：

- 是否真实存在，并且是日常场景中常见的可见物。
- 是否符合当前“场景+用途”，不混入明显跨场景、工业化、医疗化、奇幻化或过于专业的内容。
- 是否容易被欧美用户理解，避免强中式、强地域、品牌化或文化背景依赖过高的物品。
- 是否画面生成后能被一眼识别，而不是只能靠名称解释。
- 是否过于怪异、冷门、细碎或不自然。

## 2. 实质重复

审查时必须合并“本质同物”的内容。以下情况即使名称不同，也视为重复或低价值变体：

- 只换用途：如 `面包塑料袋`、`饼干塑料袋`、`零食塑料袋`，本质仍是塑料袋。
- 只换内容物：如 `马克杯糖包`、`茶包小袋`、`咖啡小袋`，若视觉上都是小袋，应控制数量。
- 只换材质：如 `塑料托盘`、`金属托盘`、`木质托盘`，若形体差异不明显，不应同时大量出现。
- 只换前缀：如 `文具收纳盒`、`桌面收纳盒`、`卡片收纳盒`，本质仍是收纳盒。
- 只换尺寸：如 `小烤盘`、`迷你烤盘`、`浅烤盘`，若差异点不明显，应保留代表项即可。

审查时优先判断“如果画出来，玩家是否会觉得这是不同物品”。如果答案是否定的，就应替换。

## 3. 命名模板密度

检查同一类别、同一池内是否出现过多相同尾词或同类结构。重点关注：

- `xxx盒`
- `xxx袋`
- `xxx盘`
- `xxx板`
- `xxx刷`
- `xxx架`
- `xxx座`
- `xxx托盘`
- `xxx收纳`
- `xxx清洁刷`
- `xxx分装盒`
- `xxx展示架`
- `xxx托座`

这些词不是完全禁用，但不能成为主要内容来源。若一个池中大量物品都依赖同一尾词，应优先替换为轮廓、功能和材质差异更明显的实物。

## 4. 找不同可玩性

每个物品都应适合做找不同差异点：

- 有清晰轮廓和可见体积。
- 适合做缺失、替换、移动、朝向或数量变化。
- 不依赖细线、小点、刻痕、碎屑、污渍、贴纸、价格签、文字或品牌标识。
- 不应太小到玩家难以观察，例如单个回形针、单颗图钉、碎纸屑等。
- 不应过薄或过扁到变化不明显，例如普通标签、纸片、小贴纸等。

如果某个物品只能靠极细微变化体现差异，应替换。

## 5. 四池边界

四个池必须保持清晰分工：

- `core_items`：当前场景的核心大件或主视觉物品。
- `support_items`：辅助中件，能丰富场景，但不抢核心。
- `visible_small_items`：小件但必须清楚可见，不能是碎屑级别。
- `scene_expansion_items`：可放置中型单物，不能是组合，不能核心化，也不能用容器模板凑数。

重点审查第四池：

- 不写 `组合`、`套装`、`一组`、`多件` 等组合概念。
- 不写第一池核心物的小变体。
- 不写过大的家具、固定装置或难以移动的空间结构。
- 不写标签、卡片、薄片、碎物等低可见内容。
- 应是可以自然放入画面、轮廓明确、单独成立的中型物。

## 6. 场景多样性

每个类别内部应有足够的物品族分布，避免单一类型堆叠。多样性审查不是一句“不要重复”，而是必须从三个维度同时检查：

- `视觉本体族`
  看画出来究竟是不是不同东西，例如书本族、瓶罐族、托盘族、支架族、软包族、工具族、纸面本体族、时间件、桌面摆件等。
- `尾词分布`
  看是否被 `盒 / 袋 / 包 / 盘 / 板 / 架 / 座 / 筒 / 器 / 桶 / 瓶 / 罐` 这类尾词统治。
- `模板骨架`
  看是否存在“不同前缀 + 相同后缀”或“不同内容物 + 相同壳体”的伪多样性，例如一串 `xxx抽取器`、`xxx托盘`、`xxx清洁刷`、`xxx样本册`。

可优先检查是否覆盖：

- 工具类
- 容器类
- 消耗品类
- 软质物类
- 装饰或功能件
- 可替换的独立实物

### 多样性复审阈值

- 任一池若 `单一尾词 > 6 条`，或 `Top 3 尾词合计 > 18 条`，记为高风险。
- `scene_expansion_items` 若 `器 / 架 / 盒 / 筒 / 座 / 托盘 / 抽取器 / 分配器 / 支架 / 挂架 / 收纳*` 合计超过 15 条，或任一单类超过 4 条，默认判为模板化过密。
- `visible_small_items` 若同一可见单位词 `一盒 / 一包 / 一筒 / 一盘 / 一套 / 一卷 / 一束 / 一叠 / 一册 / 一排` 超过 8 条，或 Top 3 单位词合计超过 24 条，必须拆散。
- 每池至少应覆盖 6 个视觉本体族；`core_items` 与 `scene_expansion_items` 优先达到 8 个以上。若某一视觉本体族达到 8 条以上，或 Top 3 视觉本体族合计达到 24 条以上，记为过密。
- 对包装、瓶罐、礼盒、零食袋这类天然高频壳体，不能只看尾词是否超额；若物品差异主要仍落在“装了什么”，而不是轮廓、使用角色、摆放方式和场景功能，仍视为伪多样性。

### 类别多样性怎么判断

- `桌面+学习`
  不能只在书、本、册、板里打转；要主动检查是否覆盖书本、文具、测量、计时、模型、图示、包类、桌面摆件等不同物品族。
- `餐桌+茶歇 / 厨房+烘焙`
  不能只用 `器 / 架 / 盒 / 筒 / 托盘` 填第四池；应补小奶壶、点心罩、黄油碟、食谱架、厚实工具、成品点心、台面服务物等自然前台物。
- `卧室+梳妆 / 浴室+洗护`
  瓶罐多是正常现象，但不能让 `瓶 / 罐` 吞掉核心池；要拆出镜子、梳具、刷具、纸品、布类、盒包、首饰收纳、个人工具、热工具等不同视觉本体。
- `商店+零食 / 节日+礼物`
  不能只用 `盒 / 桶 / 包 / 袋 / 礼盒` 假装多样；要看是否真正覆盖饼干、巧克力、糖果、坚果、饮料、试吃服务物、包装辅料、摆台陈列物等不同家族。
- `手作 / 运动 / 出行 / 园艺`
  不能只靠 `夹 / 架 / 盒 / 包 / 筒` 堆工具壳；要主动补测量、保护、承托、半成品、成品、材料、桌面本体、小设备等不同角色。

## 7. 国际化适配

默认面向欧美/国际化游戏用户：

- 避免强中式读本、器物、地域零食、节庆专属物等。
- 避免只有特定文化背景才能理解的物品。
- 避免品牌名、商标名、IP 名称。
- 保留全球常见的家居、学习、餐饮、户外、手作、运动、宠物等物品。

若某物品在欧美语境下不常见，除非该类别明确需要，否则应替换。

## 8. 数据硬规则

每次审查仍需检查基础数据质量：

- 每个池正好 50 个物品。
- 同池内无完全重复。
- 跨池无明显重复或实质重复。
- 不包含 `组合`。
- 不包含 blocked patterns。
- 不包含过小、过碎、过细或依赖文字的物品。
- Python 语法、格式和现有测试通过。

除基础数据外，审查文档还必须留下该类别的多样性结果，不得只写“重复偏多”：

- 每池 `高频尾词 Top 5`
- 每池 `高频模板骨架`
- 每池 `缺失的视觉本体族`
- 每池 `需要优先降密的尾词族 / 模板族`

## 审查输出建议

审查结果必须写入 `Game content extraction/data/审查文件/{类别} 审查文档.md`，并至少包含以下信息：

- `问题类型`：合理性 / 实质重复 / 命名雷同 / 四池边界 / 找不同可玩性 / 国际化 / 数据硬规则。
- `所在类别`：如 `桌面+学习`。
- `所在池`：如 `scene_expansion_items`。
- `问题物品`：列出需要替换的物品。
- `视觉本体族`：如笔记本族、文件夹族、托盘族、页面固定族、电子外设族。
- `问题说明`：简短说明为什么不合格。
- `修改方向`：给出更适合的物品族或具体替换建议。
- `处理状态`：待处理 / 已替换 / 保留但降级风险 / 复审通过。
- `多样性摘要`：写明该池当前最拥挤的尾词、模板骨架和缺失物品族。
- `高频尾词 Top 5`：至少记录到审查文档中，便于修改阶段回看。
- `高频模板骨架`：如 `xxx器`、`xxx架`、`xxx盒`、`内容词 + 承载壳`。
- `缺失补入方向`：明确缺的是哪几类物品，不要只写“再丰富一些”。

## 审查文件写入规则

- 审查目标明确时，创建或覆盖 `审查文件/{类别} 审查文档.md` 中对应类别的审查对象、总体结论和问题清单，避免新旧结论混在一起。
- 类别审查文档的命名必须带上具体类别：`{类别} 审查文档.md`，例如 `桌面+学习 审查文档.md`。
- 问题清单必须逐项可执行；不要只写“内容重复较多”这类无法修改的概括句。
- 同一视觉本体族的问题必须合并记录，例如把 `文件托盘`、`信件托盘`、`打印纸托盘` 归为托盘族。
- 同一尾词族和同一模板骨架的问题也必须合并记录，例如把 `糖包抽取器`、`餐巾抽取器`、`湿巾抽取器` 归为抽取器族。
- 场景常识问题必须单独记录，例如真实存在但不适合当前场景的专业设备、奇怪辅助件、低频外设。
- 若没有发现问题，也必须写入“未发现需修改项”，并说明仍完成了数量、重复、第四池和自然度检查。
- 修改阶段不得跳过对应的类别审查文档，因此审查文件中必须保留足够信息，让修改者不需要回看对话也能执行。

优先处理顺序：

1. 明显不合理或不属于场景的内容。
2. 实质重复和模板凑数。
3. 第四池边界错误。
4. 找不同可玩性差的内容。
5. 国际化适配问题。
6. 数量、格式和测试问题。
