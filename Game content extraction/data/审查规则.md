# 盲盒物品库审查规则

本文档不是审查 SOP，也不是逐步勾选表。
它是审查 `blind_boxes.py` 时必须服从的“任务宪法”：规定审查目标、边界、优先级、否决条件和审查通过标准。

审查的目的不是证明“现有内容大致能用”，而是稳定发现会让物品库变得机械、失真、伪多样或不适合找不同的根因。
只要结果仍然不自然、不稳定、不可玩，审查就不应结束；必须继续复查，直到问题被明确记录并可交给修改阶段处理。

## 1. 文档定位

- 本规则适用于全部盲盒类别。
- 本规则服务“已写库内容的发现问题阶段”，输出必须写入 `审查文件/{类别} 审查文档.md`。
- 本规则规定“什么应被判为问题、什么必须被记录、什么情况下不能放行”，不限定固定动作顺序。

## 2. 审查任务

审查的核心任务不是看格式，而是识别下面这些高风险：

- 物品不真实或不自然。
- 物品不属于当前场景前台。
- 名称不同，但画出来还是同一种东西。
- 名称只是换前缀、换内容词、换壳体。
- 第四池滑向抽取器、支架、分区盒、托盘或纯收纳壳。
- 条目不适合做找不同差异点。
- 表面上很多样，实际上只是伪多样性。

## 3. 优先级宪法

当多个判断发生冲突时，按下面顺序裁决，低优先级不得覆盖高优先级：

1. `真实与自然`
   物品首先必须是现实里常见、稳定、普通用户能自然理解的东西。
2. `场景语境`
   物品必须属于当前类别的前台语境，而不是后场、维护、仓储、安装、清洁、冷却、晾放或专业流程。
3. `物体本体`
   审查时先看它是不是一个独立成物的对象，而不是内容词、功能词或承载壳。
4. `四池角色`
   条目必须符合池的职责，尤其第四池必须是“场景补充中型物”，不是“辅助器具池”。
5. `可玩性`
   物品必须边界清楚、可单独圈选，适合做清晰差异。
6. `多样性`
   多样性必须来自物本身不同，而不是名字表面不同。
7. `数量与测试`
   数量、唯一性、自动测试和兼容结构必须通过，但这些只是底线，不是放行理由。

一句话原则：
`能过测试，不等于应判通过；像真实前台物，才配判通过。`

## 4. 审查对象

审查的不是单个名字本身，而是一个条目在下面五个层面的成立性：

- `场景层`
  它是否属于当前场景。
- `本体层`
  它是否真的是一个独立物件。
- `视觉层`
  它画出来是不是和同池已有物还是同一种东西。
- `角色层`
  它是否放错了池，尤其是否把第四池写成辅助壳体池。
- `可玩层`
  它是否能成为稳定清晰的找不同物件。

## 5. 四池定义

### `core_items`

- 审查重点：是否真的是该类别的主体物、主容器、主设备、主承载物。
- 主要风险：主体物不够典型，或被大量近义变体占满。

### `support_items`

- 审查重点：是否真的是服务核心物的常见辅助件、工具件、取用件。
- 主要风险：把核心物换个小名重写，或堆叠同类小工具壳体。

### `visible_small_items`

- 审查重点：是否真的是升级后的可见单位。
- 主要风险：仍然是单粒、单片、单张、碎屑级别，只是换了写法。

### `scene_expansion_items`

- 历史字段名保留，实际语义是“场景补充中型物”。
- 审查重点：它是否先是前台自然单物，再满足中型边界。
- 主要风险：把第四池写成抽取器、支架、分区盒、托盘、立筒、收纳座、保护壳的堆叠池。

一句话定义：
`第四池审查先看“像不像自然物”，再看“过不过边界”。`

## 6. 强否决模式

只要命中下面模式，就默认应记为问题；除非能证明它是现实中高频、稳定、独立的常用物品名，否则不得轻易放行。

### 6.1 不同前缀 + 相同后缀

例如：

- `糖包抽取器 / 餐巾抽取器 / 湿巾抽取盒`
- `文件托盘 / 信件托盘 / 打印纸托盘`
- `模具清洁刷 / 烤盘清洁刷 / 粉筛清洁刷`

这类问题的根因不是“名字相似”，而是物体本体没有真的拉开。

### 6.2 不同内容物 + 相同壳体

例如：

- `食材分区篮 / 糖珠分区盘 / 柠檬皮分区盘`
- `样本册 / 素材册 / 标本册`
- `糖果盒 / 饼干盒 / 点心盒`

这类问题的根因是“内容换了，壳没变”。

### 6.3 功能词 + 通用载体

例如：

- `隔热垫`
- `分区盒`
- `记录板`
- `整理盘`
- `展示座`

如果去掉功能、对象或内容词后，只剩通用载体，默认视为模板化高风险。

### 6.4 二级器具化

例如：

- `抽取器`
- `托架`
- `挂架`
- `收纳座`
- `保温座`
- `立筒`
- `临放盘`

尤其在第四池中，这类条目默认高风险，因为它们通常只是为另一个更小物件服务。

### 6.5 内容承载壳化

例如：

- `植物标本夹`
- `作品集画夹`
- `乐谱夹`
- `海报筒`

这类问题的根因是“真正重要的是内容本体，但命名把重点放到了壳上”。

### 6.6 后场与维护物成片出现

例如：

- 晾放、冷却、沥水、回收、封口、清洁、擦拭、维修、废弃处理物集中出现
- 偏咖啡吧台、后厨、工具房、护理台、专业制备台的物件集中出现

少量真实物件可存在，但一旦成片，就必须记录为场景错位风险。

## 7. 裸名复审原则

审查时必须做“裸名复审”。

不是机械删词，而是判断：

- 去掉对象词后，还剩什么？
- 去掉功能词后，还剩什么？
- 去掉内容物词后，还剩什么？

如果主要只剩下面这些通用壳体，默认高风险：

- `板`
- `盘`
- `垫`
- `盒`
- `包`
- `册`
- `卷`
- `袋`
- `篮`
- `筒`
- `架`
- `座`
- `托盘`
- `抽取器`
- `分区盒`
- `收纳座`
- `挂架`
- `临放盘`

关键判断不是“后缀能不能存在”，而是：

`这个名字是不是只靠后缀活着。`

## 8. 多样性标准

审查多样性时，不能只看“是否有重复词”，而要看“是否有真实差异”。

### 8.1 多样性的正确来源

应来自：

- 物体角色不同
- 视觉轮廓不同
- 摆放形态不同
- 使用阶段不同
- 物品家族不同

不应主要来自：

- 前缀不同
- 内容词不同
- 同一壳体不同
- 同一后缀换不同对象词

### 8.2 多样性复审必须同时看三条线

- `视觉本体族分布`
- `高频尾词分布`
- `模板骨架密度`

即使尾词没有超额，只要仍然是一串“内容物 + 通用壳体”或“对象词 + 同一器具”，也必须记为问题。

### 8.3 多样性风险阈值

- 任一池若 `单一尾词 > 6 条`，或 `Top 3 尾词合计 > 18 条`，记为高风险。
- `scene_expansion_items` 若 `器 / 架 / 盒 / 筒 / 座 / 托盘 / 抽取器 / 分配器 / 支架 / 挂架 / 收纳*` 合计超过 15 条，或任一单类超过 4 条，默认记为模板化过密。
- `visible_small_items` 若同一可见单位词超过 8 条，或 Top 3 单位词合计超过 24 条，记为单位词过密。
- 每池至少应覆盖 6 个视觉本体族；`core_items` 与 `scene_expansion_items` 优先达到 8 个以上。若某一视觉本体族达到 8 条以上，或 Top 3 视觉本体族合计达到 24 条以上，记为本体族过密。
- 对包装、瓶罐、礼盒、零食袋等天然高频壳体，不能只看尾词是否超额；若差异主要仍落在“装了什么”，仍视为伪多样性。

## 9. 第四池专项审查

第四池是最容易出问题的区域，必须单独从下面几条判断：

### 9.1 第四池应通过什么

- 是现实中稳定存在的中型单物。
- 比 `visible_small_items` 更清楚，比 `core_items` 更弱主角。
- 普通玩家一眼就能认出它是什么。
- 不依赖解释“用来放谁、固定谁、抽取谁”才能成立。

### 9.2 第四池应拦截什么

- `组合`
- 第一池核心物的近义变体、材质变体、尺寸变体、形状变体
- 大型家具、陈列架、推车、柜子等大主体
- 功能词 + 通用载体的临时造词
- 二级器具化命名
- 内容承载壳化命名
- 为了过中型边界而故意补后缀的壳体词

### 9.3 第四池的常见错觉

- “它能通过中型边界测试，所以没问题”
- “它不是重复名字，所以算多样”
- “它确实存在，所以属于当前场景”

以上三条都不成立。
第四池审查一律以“前台自然单物”优先。

## 10. 可玩性标准

审查时，条目必须适合做找不同差异点：

- 有清晰轮廓和可见体积。
- 适合做缺失、替换、移动、朝向或数量变化。
- 不依赖文字、标签、价格签、刻度、细线、小点、碎屑、污渍、贴纸、发光、反光、透明、高光或阴影变化。
- 不应过小、过薄、过碎或极度依赖局部细节。

如果某个物品只能靠非常细微的变化成立，就应记为可玩性风险。

## 11. 国际化适配

默认面向欧美 / 国际化找不同用户。

审查时，除非类别明确要求，否则应记为问题的情况包括：

- 强中式、强地域、强课程体系绑定物品
- 只有特定文化背景才能自然理解的物品
- 品牌名、商标名、IP 名称

中文只是输出语言，不等于物品本体可以是强地域专属。

## 12. 数据底线

审查不能只停留在内容判断，也必须覆盖基础结构：

- 每池正好 50 条
- 池内唯一
- 跨池无机械精确重复
- 不包含 `组合`
- 不命中 blocked patterns
- 不引入过小、过碎、过细、依赖文字的物品
- Python 语法、兼容结构和现有测试可通过

但这些只是底线，不是放行理由。

## 13. 自主复查义务

只要出现下面任何一种情况，审查都不应停在“基本可用”：

- 名字虽然不同，但读起来明显像一串模板
- 虽然过了数量和测试，但场景明显不自然
- 第四池明显滑向壳体池或器具池
- 多样性看起来只是换壳体、换内容、换前缀
- 普通人一眼不会自然这样命名或理解

必须继续复查，并把问题写清楚，让修改阶段可以直接处理根因。

一句话要求：
`如果读起来像 AI 在为过规则而造词，就说明审查还没完成。`

## 14. 审查通过标准

一个类别的审查只有在下面条件同时满足时，才算“审查完成”：

### 14.1 基础结构

- 四池数量、唯一性、跨池精确重复和兼容结构已检查。

### 14.2 问题发现充分

- 不是只写“重复较多”这种模糊结论，而是明确指出问题族、问题模式和场景风险。

### 14.3 多样性结果明确

- 已记录该类别至少以下结果：
  - `高频尾词 Top 5`
  - `高频模板骨架`
  - `缺失的视觉本体族`
  - `需要优先降密的尾词族 / 模板族`

### 14.4 第四池风险明确

- 已明确第四池是否存在二级器具化、内容承载壳化、功能词造词、核心变体复写或模板过密。

### 14.5 输出可执行

- 审查文档足以让修改阶段在不回看对话的前提下继续工作。

## 15. 审查文档要求

每个类别的审查输出都必须写入：

- `Game content extraction/data/审查文件/{类别} 审查文档.md`

至少应能回答：

- 这个类别主要问题是什么
- 哪些条目必须替换
- 它们各自属于什么问题模式
- 第四池的根因是什么
- 当前最拥挤的尾词和模板是什么
- 缺的是哪些物品族
- 修改应朝什么方向补

审查文档不是流水账，而是修改阶段的任务单。

## 16. 类别提醒

这些提醒不是白名单，只是常见偏差校正：

- `桌面+学习`
  不要只在书、本、册、图册、挂图、夹壳里横向平移。
- `餐桌+茶歇 / 厨房+烘焙`
  不要把第四池审查放松成 `器 / 架 / 盒 / 筒 / 托盘 / 垫` 的尾词检查；重点要看它是不是前台自然物。
- `卧室+梳妆 / 浴室+洗护`
  瓶罐多是正常现象，但不能让瓶罐壳和收纳壳吞掉整池多样性。
- `商店+零食 / 节日+礼物`
  重点拦截 `盒 / 桶 / 包 / 袋 / 礼盒` 造成的伪多样性。
- `手作 / 运动 / 出行 / 园艺`
  重点拦截 `夹 / 架 / 盒 / 包 / 筒` 造成的工具壳体扩写。

## 17. 最终判语

合格审查不是“把格式问题查完”，而是：

- 真正指出不自然的地方；
- 真正识别伪多样性；
- 真正区分物体本体和承载壳；
- 真正把第四池从器具池里拉出来。

如果审查结论仍然会让修改阶段继续沿着“换前缀、换壳体、过测试就行”的方向走，那这次审查就视为未通过，必须继续复查并补齐判断。
