# 盲盒物品库修改规则

本文档不是 SOP，也不是逐步操作表。
它是修改 `blind_boxes.py` 时必须服从的“任务宪法”：规定目标、边界、优先级、否决条件和验收标准。

当自动测试通过但结果仍然别扭、机械、伪多样或场景不自然时，必须继续复查并继续修改；不得把“已经过测试”当作停止条件。

## 1. 文档定位

- 本规则适用于全部盲盒类别。
- 本规则服务“审查后修改”阶段：先有 `审查规则.md` 的问题归因和 `审查文件/{类别} 审查文档.md` 的问题记录，再按本文件完成替换与复查。
- 修改完成后的复审记录，继续回写同一份类别审查文档；固定结构以 `审查文件.md` 为准。
- 一轮替换完成后，再读 `复查清单.md` 做收尾确认。

## 2. 核心任务

修改的目的不是把一批不合格名称换成另一批更会过规则的名称，而是：

- 让物品更真实。
- 让场景更自然。
- 让四池分工更清楚。
- 让多样性来自“物本身不同”，而不是“前缀不同、壳体不同、后缀不同”。
- 让每个条目都适合找不同游戏中的单点差异。
- 让旧问题族真正消失，而不是从一批坏模板词滑到另一批坏模板词。

## 3. 优先级宪法

当多个标准冲突时，按下面顺序裁决，低优先级不得推翻高优先级：

1. `真实与自然`
   物品必须是现实里常见、稳定、普通用户能自然理解的东西。
2. `场景语境`
   物品必须属于当前类别的前台语境，而不是后场、维护、安装、仓储、回收、冷却、晾放或专业操作阶段。
3. `物体本体`
   物品必须首先是一个“独立成物”的对象，而不是内容物、功能词或小物的承载壳。
4. `四池角色`
   条目必须符合所在池的职责，尤其第四池必须是“场景补充中型物”，不是“辅助器具池”。
5. `可玩性`
   条目必须边界清楚、第一眼可识别、适合作为单点差异。
6. `多样性`
   多样性必须来自物体角色、轮廓、使用场景、摆放形态和本体族差异，不得来自换前缀、换内容词或换通用壳体。
7. `测试与配额`
   自动测试、数量、唯一性、尾词配额和兼容字段必须通过，但它们只是底线，不是生成方向。

一句话原则：
`能过测试，不等于应该入库；像真实场景物，才配入库。`

## 4. 四池定义

### `core_items`

- 写该类别最能代表场景与用途的主体物、主容器、主设备、主承载物。
- 不写微小配件、后场辅助物、纯功能壳体。

### `support_items`

- 写服务主体物使用的常见辅助件、工具件、替换件、取用件。
- 不写单个极小配件，不把同一基础对象机械换名重写。

### `visible_small_items`

- 写原本偏小、但已升级成清晰可见单位的条目，如一盒、一包、一筒、一盘、一套、一卷、一束、一叠、一册、一排。
- 不写单粒、单张、单片、单根、碎屑、小点、细线、标签、价格签。

### `scene_expansion_items`

- 历史字段名保留，实际语义是“场景补充中型物”。
- 它应当是普通玩家一眼能认出的、能独立出现在画面前台的自然物件。
- 它以单个中型实物为主；全池只允许极少量简单组合，最多 5 条，且每条组合最多 3 个内容。
- 简单组合只接受一眼能认出的同类或近邻小组，如 `贝壳组合`、`点心组合`；不接受复杂拼装、散堆拼盘或靠说明才能成立的组合。
- 它是弱主角，不是小物的抽取器、支架、托架、收纳座、保护座、分区盒或临放盘。
- 它不把架子类、箱子类、柜子类、支架类当作第四池的常规来源。
- 它必须先是“物”，再是“功能”；先能独立成立，再谈中型边界。

一句话定义：
`第四池主要写前台自然中型实物，只允许极少量简单组合，不写为了过边界而临时造出来的壳体。`

## 5. 准入标准

任一候选条目，只有同时满足下面标准，才允许保留：

### 5.1 现实自然

- 普通用户会自然承认“这就是一个真实存在的东西”。
- 不依赖专业知识、课程体系、吧台流程、工坊流程或维护流程才看得懂。

### 5.2 场景前台

- 它会自然出现在当前类别画面的前台。
- 它不是清洁、回收、冷却、封口、晾放、维护、储运、安装或后处理阶段物。

### 5.3 独立成物

- 它本身就是物件，而不是“内容物 + 一个通用壳体”。
- 它本身就是物件，而不是“功能词 + 一个通用板 / 盘 / 垫 / 盒 / 册 / 卷”。
- 它本身就是物件，而不是“另一个更小物品的附属取用 / 承托 / 固定器具”。

### 5.4 第一眼可识别

- 脱离文字说明也能被当成一个明确物件。
- 不靠文字、图标、刻度、标签、小细节或非常细的局部结构成立。
- 不需要额外解释“它到底属于什么类别的东西”。
- 如果必须补一句“它其实是某种教具 / 演示器材 / 手作工具 / 课程用品”，默认不够自然。

### 5.5 池内角色清楚

- 它没有偷跑成核心物，也没有退化成微小配件。
- 尤其第四池，不得只是把前三池已有物套进另一个壳。

## 6. 需要优先替换的高风险模式

本节只保留修改阶段最常用的模式摘要；详细问题归类沿用 `审查规则.md`。

### 6.1 不同前缀 + 相同后缀

例如 `糖包抽取器 / 餐巾抽取器`、`文件托盘 / 信件托盘`。
根因不是“名字像”，而是物体本体没有真的拉开。
如果只是给旧物前面加场景词、口味词或材质词，主词没变，也按这一类处理。
例如 `磅蛋糕 -> 柠檬磅蛋糕`、`茶包 -> 篝火茶包`、`燕麦棒 -> 徒步燕麦棒`。

### 6.2 不同内容物 + 相同壳体

例如 `糖果盒 / 饼干盒 / 点心盒`、`样本册 / 素材册 / 标本册`。
根因是“内容换了，壳没变”。

### 6.3 同模板横向平移

例如 `摄影机支架 / 面罩支架 / 样本瓶架`、`作品记录本 / 项目记录本 / 行数记录本`。
如果变化主要落在前缀说明，而稳定不变的是同一个后缀本体或同一职能骨架，就按同模板伪多样性处理。

特别强调：

- `只要主词没变，默认不算修复`
- `同族平移替换不算修复`
- `作品记录本 -> 项目说明本` 不算修改成功
- `海流记录板 -> 潜点提示板` 不算修改成功
- `课程记录板 -> 试卷分区架 -> 学习提示器` 这类弱本体链条横跳，仍视为假修复

### 6.4 说明成分 + 通用载体

例如 `隔热垫`、`记录板`、`展示座`、`摄影机支架`、`电路实验板`。
如果去掉功能、对象、流程、用途或被服务对象后，只剩通用载体，默认高风险。

### 6.5 二级器具化

例如 `抽取器 / 托架 / 支架 / 展示架 / 底座 / 收纳座 / 立筒 / 临放盘`。
尤其第四池中，这类条目通常只是为另一个更小物件服务，不应作为主来源。

### 6.6 内容承载壳化

例如 `植物标本夹 / 作品集画夹 / 乐谱夹 / 海报筒`。
这类名字把真正重要的内容本体塞进了承载壳里，画面重点不稳定，也容易机械扩写。

### 6.7 解释型命名

例如 `电路实验板 / 月相演示球 / DNA结构摆件 / 植物压花板`。
这类词更像说明词、用途词或课程词，而不是普通玩家自然会说出的前台物名。

### 6.8 后场与维护物

例如晾放、冷却、沥水、回收、封口、清洁、维修等流程物，或偏后厨、吧台、工具房、专业制备台的物件成片出现。
少量真实条目可存在，但不得成为某池的主要来源。

## 7. 裸名复审与替换方向

修改时必须做“裸名复审”。
去掉对象词、被服务对象词、抽象职能词、功能词、动作词或内容物词后，如果名字主要只剩下 `板 / 盘 / 垫 / 盒 / 包 / 册 / 卷 / 袋 / 篮 / 筒 / 架 / 座 / 托盘 / 抽取器 / 分区盒 / 收纳座 / 挂架 / 临放盘` 这类通用壳体，默认高风险。

替换时遵守下面原则：

- 不在同一模板族里平移换词。
- 优先跳出原后缀家族，换到别的真实物族。
- 第四池优先补前台自然单物，不补为了过边界临时造出来的壳体词。
- 第四池若要写组合，只能写极少量简单小组合；不要把“组合”当成放宽边界后的新模板池。
- 如果某类已经有一个抽象职能文书代表项，不再继续在同族里横向扩写。
- 回填时先想“真实前台物族缺什么”，不要先想“什么后缀更容易过边界”。
- 如果新结果一眼能读成另一串 `xx板 / xx架 / xx器 / xx机 / xx本 / xx册`，默认不是收尾，而是回到问题族层面重做。

## 8. 多样性复审

多样性不是“把名字改得不一样”，而是让物体真的不一样。
每轮修改后，至少同时复看：

- `视觉本体族分布`
- `高频尾词分布`
- `模板骨架密度`
- `解释成本是否仍然偏高`

如果复查后仍能明确指出下面任一情况，就必须继续替换：

- 单一尾词仍明显过密
- 同一模板骨架仍成片出现
- 仍然存在“不同前缀 + 相同后缀”的伪多样性
- 仍然存在“不同内容物 + 相同壳体”的伪多样性
- 虽然名字变了，但视觉本体族没有真正拉开
- 同后缀家族仍然一眼读成一串
- 仍然存在“需要靠学科 / 课程 / 功能解释才能成立”的条目
- 旧问题族虽然消失了，但新结果立刻长成另一条弱模板家族

具体阈值沿用 `审查规则.md` 中的多样性风险阈值，不在本文件重复抄写。

## 9. 测试与边界的关系

自动测试、数量和字段兼容是必须满足的底线，但它们不能反过来支配命名方向。

特别强调：

- `test_scene_expansion_items_keep_medium_scene_fragment_boundary` 只是底线测试。
- 测试允许的后缀，不是推荐造词模板。
- “能过中型边界”不等于“自然”“真实”“应该入库”。
- 如果一轮修改主要靠一批边界友好后缀通过，默认仍是失败，不得据此收尾。

任何条目，只要出现下面情况之一，就算测试通过也必须继续修改：

- 读起来像临时拼出来的壳体词
- 去掉前缀后只剩通用载体
- 去掉被服务对象词后只剩 `支架 / 底座 / 托架 / 展示架 / 挂架`
- 去掉说明成分后只剩 `板 / 垫 / 盘 / 盒 / 册`
- 一眼看不出它到底是什么物
- 明显更像后场流程物
- 明显只是为了通过测试才带上某个后缀

## 10. 自主复查义务

当结果有问题时，AI 不能停在“已经修改完”。
完成一轮替换后，必须读 `复查清单.md` 做收尾复查；如果仍存在场景不自然、第四池像辅助器具池、伪多样性明显、同模板条目成片、普通人不会自然这样命名等情况，就继续进入下一轮。

这是一条闭环义务，不是一次性检查：
`替换 -> 复查 -> 发现问题 -> 再替换 -> 再复查`

一句话要求：
`如果你自己读着都像是在硬造词，就还没有改完。`

补充要求：
`如果只是把坏家族 A 换成坏家族 B，就还不算优化，只算换壳。`

## 11. 验收标准

修改完成后，至少必须同时满足下面条件，才算通过：

### 11.1 基础结构

- 每池正好 50 条。
- 池内唯一。
- 与其他池无机械精确重复。
- 兼容字段和运行时映射不被破坏。

### 11.2 真实性

- 条目是真实可见的物，不是抽象名、过程名、壳体名或临时功能名。
- 普通用户第一眼能把它当成一个物件。
- 普通用户不需要额外说明“这其实是某种教具 / 演示器材 / 压花工具”才知道它是什么。

### 11.3 场景自然度

- 条目属于当前类别前台。
- 不成片滑向后场、维护、清洁、冷却、封口、安装、仓储或专业流程。

### 11.4 四池角色

- 第四池不再像抽取器池、支架池、托盘池、分区盒池。
- 第四池的条目首先是“场景物”，不是“别的物的辅助壳”。
- 第四池不再成片出现 `对象词 + 支架 / 底座 / 托架 / 展示架` 这类服务壳体。
- 第四池不再成片出现 `说明词 + 板 / 垫 / 盘 / 盒 / 册` 这类虚化弱本体。

### 11.5 多样性

- 不存在明显的“不同前缀 + 相同后缀”成片堆叠。
- 不存在明显的“不同内容物 + 相同壳体”成片堆叠。
- 多样性主要来自物体本体差异，而不是壳体差异。
- 修改后的结果重新复查时，不再触发需要继续整改的多样性风险。

### 11.6 可玩性

- 适合缺失、替换、移动、朝向、数量等清晰差异。
- 不依赖文字、标签、刻度、细小结构、发光、反光、透明或阴影。

### 11.7 自动化底线

- 内容测试通过。
- 第四池中型边界通过。
- 配额与唯一性检查通过。

## 12. 复审记录要求

修改完成后，对应类别审查文档必须补齐下面三段：

- `本次修改记录`
- `修改后复审`
- `剩余风险与暂留理由`

并在这三段之下至少回答：

- 本次改了什么
- 本次主要压掉了哪类伪多样性
- 本次替换方向是什么
- 还有哪些剩余风险
- 为什么这些剩余项暂时保留

审查文档不是流水账，而是“本次判断为何成立”的依据；不要另起临时总结文件替代类别审查文档。

## 13. 类别提醒

这些提醒不是白名单，只是常见偏差校正：

- `桌面+学习`：不要把第四池做成书册、图册、挂图、夹壳的横向平移，也不要滑向“学科教具名 + 板 / 球 / 器 / 摆件”。
- `餐桌+茶歇 / 厨房+烘焙`：不要把第四池做成抽取器、托座、纸品壳体、后场清洁物或“服务动作 + 载体”的伪自然写法。
- `卧室+梳妆 / 浴室+洗护`：不要让盒、筒、架、盘、托盘、分格壳、瓶罐壳或纯收纳壳吞掉整池多样性。
- `商店+零食 / 节日+礼物`：不要把第四池做成礼盒、零食盒、外带袋、分配器或包装壳体横向换词。
- `手作 / 园艺 / 运动 / 出行`：不要把第四池做成夹、架、盒、筒、袋、护套和说明性壳体的机械扩写。

## 14. 最终判语

合格修改不是“看起来像做了很多替换”，而是：

- 用户一看就觉得更自然
- 场景一看就更像真实前台
- 条目一看就不是硬造词
- 多样性一看就不是换壳体

如果结果仍然让人产生“这名字像 AI 为了过规则临时拼出来的”感觉，就视为未通过，必须继续修改。
