# Spec Handoff: `scene_expansion_items` 中号场景片段化

## 1. 背景

当前 `Game content extraction/data/blind_boxes.py` 中 20 个盲盒场景的 `scene_expansion_items` 大量使用柜、车、落地架、陈列架、工作台、桌、棚等大件名词。该池在兼容视图中与 `core_items` 一起进入运行时 `large`，因此这些词条会直接增加大型物品抽取时的空间占用压力。

前序 brainstorm 已排除两个不理想方向：

- **不做简单小型容器替换**：盒、篮、提篮、收纳包容易与 `core_items` / `support_items` 里的收纳盒、整理篮、托座混淆。
- **不做小卡片替换**：小卡片、小标签、单枚价格签、编号牌等主体太小，容易退化为 `visible_small_items` 或细碎差异点。

本交接文档将最终方向固定为：**中号场景片段池**。

## 2. 目标

将 20 个场景的 `scene_expansion_items` 从“大型家具/收纳设施”改写为“中号场景片段”：

- 比单张卡片、标签、小票、价格签更大。
- 比柜、车、落地架、桌、工作台、棚更小。
- 主体必须第一眼可见、边界清楚、可单独圈选。
- 通过活动结果、任务面、铺垫界面、成组阵列、完整成果来扩展场景感。
- 保持四池结构、场景编号、运行时兼容映射不变。

## 3. 非目标

- 不新增第五池，不引入 `scene_anchor_items`。
- 不改变 `BLIND_BOX_SCENE_ENTRIES` 的 20 个场景入口和编号语义。
- 不改变 `BLIND_BOX_COMPATIBILITY_MAPPING` 的四栏映射。
- 不改 UI、历史抽取逻辑、动物池或表情逻辑。
- 不把 `scene_expansion_items` 简单改成 `support_items` 或 `visible_small_items` 风格。

## 4. 内容模型

### 4.1 推荐五族群

每个场景的 50 条 `scene_expansion_items` 建议围绕以下五族群生成，可按场景适度调整比例：

1. **活动结果物**
   - 已完成或正在完成的活动结果。
   - 示例：待批改作业叠、包装到一半的礼物组、修剪好的花枝束。

2. **中号任务面**
   - 板、垫、展开页、记录面、步骤面、检查面。
   - 示例：复习板、步骤板、记录板、检查板、提示板。

3. **铺垫界面物**
   - 可作为场景承载面的中号平面物。
   - 示例：操作垫、餐布角、毛巾垫、防滑垫、防潮垫。

4. **成组阵列物**
   - 多个可见物组合成一个明确阵列或区域。
   - 示例：训练标志盘阵列、杯垫阵列、玩沙工具排列组、补给瓶组。

5. **完整成果物**
   - 半成品或成品本身成为场景片段。
   - 示例：半完成拼图板、织到一半的围巾片、装饰到一半的小蛋糕盘。

### 4.2 尺度边界

推荐主体：

- `板`
- `垫`
- `布`
- `展开页`
- `拼图板`
- `操作面`
- `记录板`
- `步骤板`
- `检查板`
- `提示板`
- `成组阵列`
- `半成品块`
- `成果盘`
- `组合盘`
- `底板`
- `排列组`

禁止作为独立主体：

- `小卡`
- `卡片`
- `标签`
- `签`
- `价格签`
- `编号牌`
- `铭牌`
- `名牌`
- `小票`
- `书签`
- `票卡`
- `单枚`
- `单张`

信息类内容可以作为更大主体的附属细节，但不能作为主物：

- `价格签` -> `待补货价格板`
- `标签牌` -> `标签排版板`
- `课程卡` -> `复习板` / `课程安排板`
- `植物铭牌` -> `育苗记录板`
- `小票` -> `收银记录板`
- `书签` -> `摘录页组`

### 4.3 大件强降权词

`scene_expansion_items` 中应强降权或删除以下大件倾向词：

- `柜`
- `车`
- `推车`
- `整理车`
- `收纳柜`
- `边柜`
- `抽屉柜`
- `落地`
- `落地架`
- `陈列架`
- `展示柜`
- `货架`
- `书架`
- `桌`
- `工作台`
- `作业台`
- `棚`
- `凳`

例外：若个别场景确实需要中型承载物，每场景最多保留少量，且应优先使用不占地的中号主体，例如 `操作垫`、`记录板`、`成果盘`、`排列组`。

## 5. 变更范围

### 必改文件

- `Game content extraction/data/blind_boxes.py`
  - 重写 20 个场景的 `scene_expansion_items`。
  - 每个场景仍保持 50 条唯一项。
  - 保持 `core_items`、`support_items`、`visible_small_items`、`scene_expansion_items` 四池 key 不变。

### 可能同步文件

- `Game content extraction/agents.md`
  - 如实现接受“中号场景片段”语义，应更新“写库验收”中关于场景扩展物的说明。
  - 建议将“场景扩展物必须中等以上且无需画面已有对象成立”细化为“以中号场景片段为主，不使用大型家具/收纳设施，也不使用小卡片/标签作为主体”。

- `agents.md`
  - 仅当需要沉淀稳定规则时更新；保持精简。

- `README.md`
  - 通常不需要更新，除非面向用户的盲盒库语义说明发生变化。

- `.gitignore`
  - 本任务预计不需要更新。

## 6. 推荐实施顺序

1. **先做 3 个 pilot 场景**
   - `桌面+学习`
   - `公园+野餐`
   - `海底+潜水`

2. **人工检查 pilot**
   - 是否仍有柜/车/落地架等大件。
   - 是否仍有小卡片/小标签/单枚价格签等过小主体。
   - 是否和 `support_items` / `visible_small_items` 混淆。

3. **扩展到 20 个场景**
   - 为每个场景列出 8-12 个专属词根。
   - 按五族群生成 50 条。
   - 保持每条第一眼可见、边界清楚、可单独圈选。

4. **补质量校验**
   - 至少用现有 unittest 验证四池 50 条唯一项。
   - 建议新增或扩展内容质量测试，检查大件强降权词和过小主体词。

5. **同步文档**
   - 根据实际实现，更新 `Game content extraction/agents.md` 的写库验收规则。

## 7. 验收标准

### 数据结构

- 20 个场景仍全部存在。
- 每个场景仍有四池：
  - `core_items`
  - `support_items`
  - `visible_small_items`
  - `scene_expansion_items`
- 每个池仍为 50 条唯一项。
- `BLIND_BOXES` 和 `BLIND_BOX_COMPATIBILITY_MAPPING` 的现有兼容行为不变。

### 内容质量

- `scene_expansion_items` 主体不再以柜、车、落地架、陈列架、工作台、桌、棚为主。
- `scene_expansion_items` 主体不使用小卡片、小标签、单枚价格签、编号牌、小票、书签等过小物。
- 信息类词汇如价格、标签、编号、记录可以出现，但必须升级为中号主体，例如 `价格板`、`记录板`、`排版板`、`展开页组`。
- 每条都是可见、块状或成区域的中号物，不依赖光影、反射、透明、发光、细线或微型痕迹。
- 与 `support_items` 的区别：不是单个工具/托座/小配件，而是能表达场景活动或状态的中号片段。
- 与 `visible_small_items` 的区别：不是散落小件或数量很小的碎片，而是板面、垫面、阵列、成果盘、半成品区域。

### 回归验证

在 `Game content extraction/` 下运行：

```powershell
python -B -m py_compile '内容抽取.py'
python -B -m py_compile 'image_fetcher_ui.py' 'file_batch_renamer.py'
python -B -m unittest discover -s . -p 'test_*.py'
```
