# Implementation Plan: Game Content Expression Window

## 1. Requirements Summary

为 `Game content extraction` 新增独立人物表情内容抽取窗口。用户粘贴 `组图 23 表情前置.md` 产出的表情组文本后，工具读取 `极性`、`具体表情`、`单人/多人`，从 `../组图 23 表情库.md` 按 `极性 + 具体表情 + 单人/多人` 查找眉/眼/嘴模板，并把模板追加到原文 `具体表情:` 字段后。

MVP 必须覆盖：

- 独立 `Toplevel` 窗口，不改变现有盲盒/动物抽取主输入语法。
- 支持单行和多行表情组文本。
- 支持指定模板编号；单人合法编号为 1-4，多人合法编号为 5-8。
- 支持随机策略作为可选项，但验收样例必须能指定第 4 条。
- 原文局部回填，只增强 `具体表情:` 字段，其他字段顺序和内容尽量保持原样。
- 支持同一输入中的多组 `极性:` 表情组，逐组匹配并回填。
- 重复抽取同一文本时不得无限叠加多份 `眉/眼/嘴` 模板。
- 保留 `[目标物]`、`[证据物]`、`[对方人物]`、`[剧情食物]`、`[剧情小物]`，不自动替换。
- 不接入 `draw_history.json`，不影响现有历史和重置按钮。
- 中文错误提示覆盖空输入、字段缺失、类别不存在、极性错配、模板编号越界、表情库缺失或结构异常。

## 2. Architecture Decisions

- 在 `BlindBoxExtractor` 内按现有风格新增方法，避免引入新框架或大规模拆分。
- 纯逻辑优先：表情库路径解析、Markdown 解析、字段解析、模板选择、局部回填先实现为可单独调用的方法。
- UI 后接入：主界面只新增一个入口按钮，表情抽取使用独立 `Toplevel`，窗口内部维护自己的输入和输出控件。
- 表情库继续作为单一事实源读取，不把模板复制进 `data/` 或 `draw_history.json`。
- 打包阶段只处理必要 `datas` 和运行时路径，不改变用户维护 Markdown 的方式。

## 3. Task Breakdown

1. `IMPL-1`: 表情库路径与 Markdown 解析核心。
2. `IMPL-2`: 表情组字段解析、模板选择和原文局部回填。
3. `IMPL-3`: 独立 `Toplevel` UI 接入。
4. `IMPL-4`: 打包路径与 PyInstaller spec 更新。
5. `IMPL-5`: 验收样例、回归检查和轻量测试脚本。

## 4. Implementation Strategy

按顺序执行。`IMPL-1` 和 `IMPL-2` 是核心依赖，完成后即可用纯文本样例验证。`IMPL-3` 只消费前两步方法，不重写业务逻辑。`IMPL-4` 在功能可运行后处理资源路径。`IMPL-5` 收尾验证主抽取流程未被破坏。

推荐执行命令：

```text
$workflow-execute --session WFS-game-content-expression-window
```

执行前建议先人工确认计划，因为该功能会改动单文件 tkinter 主程序和 PyInstaller 配置。

## 5. Risk Assessment

- 低风险：功能可作为独立窗口接入，不需要改变既有主抽取流程。
- 中等注意：`内容抽取.py` 已是单类文件，新增方法要命名清晰，避免让主逻辑更难读。
- 中等注意：PyInstaller `console=False` 打包后，根目录 Markdown 相对路径可能和源码运行路径不同。
- 中等注意：字段解析需要兼容中文冒号、英文冒号、同一行文本和多行文本；实现必须局部回填，不能重排整段文本。
- 中等注意：多组输入和重复增强都容易造成局部回填 span 错误，执行时必须把这两类样例纳入验证。

## 6. Verification Plan

- 用验收样例验证：`负向 + 困惑 + 单人 + 模板编号 4` 必须追加：
  `眉：一侧眉尾抬起，另一侧眉尾压平；眼：一侧眼撑开看着[目标物]，另一侧眼半垂；嘴：一侧嘴巴闭住下压，另一侧嘴角收紧。`
- 验证同一文本的其他字段、字段顺序和禁用区域原文保持不变。
- 验证多组输入能逐组增强，且重复点击不会堆叠多份眉/眼/嘴。
- 验证主界面原有输入样例 `1,5,地面动物,无大型物品,中型物品+1` 仍可解析。
- 验证表情抽取不会写入 `draw_history.json`。
- 验证源码运行和 PyInstaller 打包运行都能找到 `组图 23 表情库.md`。
