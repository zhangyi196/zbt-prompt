# Agents Guide

本文件为进入该仓库工作的 AI 代理提供项目背景、文件职责与编辑约束。适用范围为仓库根目录下的全部 Markdown 提示词文档。

## 项目概览

本仓库维护一组用于卡通休闲游戏生产流程的中文提示词规范，重点服务于“找不同”关卡设计、差异点生成、场景候选物审核与摆放。

当前仓库不是代码项目，没有构建脚本、测试框架或运行入口。主要工作是维护 Markdown 文档中的提示词规则、输出格式、合规自检与版本说明。

## 文件职责

- `主图 第二步 .md`
  - 角色：场景候选物审核与摆放设计师。
  - 用途：基于用户提供的 Target 底图和 0-6 个候选内容，完成场景扫描、候选审核、有限修改与最终摆放。
  - 核心约束：只在用户完全未提供候选时才可自行补充 6 个候选；只要用户提供至少 1 个候选，就不得补全、追加或猜测新候选。

- `组图 4.md`
  - 角色：找不同游戏关卡设计润色师。
  - 用途：基于 Group 2 Image 设计 18 个全新的差异点，用于生成 Group 3。
  - 核心约束：九宫格每区固定 2 个点；总数刚好 18 条；严格满足类型配额 3 / 6 / 3 / 3 / 3；禁止灯光、发光、明暗、反光、高光、阴影等相关变化与词汇。

- `组图 23.md`
  - 角色：Ref-A / Ref-B / Target 的差异迁移设计器。
  - 用途：分析 Ref-A 到 Ref-B 的真实可见变化，并将变化逻辑迁移到 Target 中，输出核心差异 List 1 和辅助差异 List 2。
  - 核心约束：禁止脑补；只绑定 Ref 中真实可见变化；List 1 固定 16 条，List 2 固定 16 条；输出固定六个板块；同一人物的面部表情区与衣领/领带/颈部区、头发/头饰区、耳饰/耳侧饰品区互斥，且该互斥规则高于 Ref 复刻。

## 编辑原则

1. 保持中文提示词风格，避免把规范改写成代码注释式说明。
2. 保留现有 Markdown 层级、编号、粗体重点、`[CRITICAL]` 标记和固定输出格式。
3. 修改规则时优先做局部调整，不要无关重排整篇文档。
4. 不要删除硬性禁令、自检清单、数量配额、输出格式示例或固定句式，除非用户明确要求。
5. 若调整了行为边界，必须同步检查相关的输入规则、工作流程、输出格式和合规性自检，避免前后矛盾。
6. 文档应保存为 UTF-8 编码。读取文件时如果出现乱码，应显式使用 UTF-8。

## 关键业务不变量

- 禁止脑补不可见对象、画外空间、遮挡后区域或不稳定身份对象。
- 差异设计必须遵守“一圈一物一变化”或“单一不同点”原则。
- 不修改人物骨架、动作姿态、头身朝向、四肢动作或支撑关系。
- 对 `组图 23.md`，同一人物若修改面部表情，不得再修改其衣领/领带/颈部、头发/头饰、耳饰/耳侧饰品区域；若不修改面部表情，才可选择这些人物附属区域。
- 对 `组图 23.md`，表情嘴部可以承载 Ref 明确可见且剧情需要的物体或状态，如嘴巴塞满食物、嘴里叼着物体、嘴角夹着纸条、嘴边露出碎屑；不得脑补来源、动作过程或额外剧情。
- 不把小变化升级成灾难、阴谋、案件、长期危机或大型事故。
- 不做灯光、发光、明暗、阴影、高光、反光、滤光、投影、镜面成像等变化。
- 不整体修改大面积背景、大物体主体或大面积表层。
- 位置字段只用于定位，避免颜色、材质、纹理、风格、明暗等修饰词。
- 若规则冲突，优先保留硬性禁令、数量配额、输出格式和用户提供内容边界。

## 工作流程

代理开始修改前应先完成：

1. 阅读目标文件的当前版本号、角色、输入规则、核心约束和输出格式。
2. 明确本次修改属于哪类：补充规则、收紧禁令、调整输出格式、修正文案、消除矛盾或版本升级。
3. 搜索同类规则在全文中的重复位置，确保相关段落同步一致。
4. 只编辑与本次需求直接相关的段落。
5. 修改后通读受影响章节，检查是否出现数量、格式、术语或优先级冲突。

## 版本与命名

- 文件标题中带有版本号，例如 `v1.7.9.7`、`v1.3.1`、`v2.3`。
- 若用户要求发布新版、记录版本变化或修改属于明显规则升级，应更新标题版本号。
- 若只是轻微错字、格式或局部措辞修正，可不主动改版本号，除非用户另有要求。
- 文件名包含中文和空格，命令行操作必须使用引号或 `-LiteralPath`。

## 输出格式维护

维护提示词时尤其注意：

- 固定板块数量不得随意改变。
- 列表条数要求必须明确，例如固定 16 条、固定 18 条、5-6 个落点等。
- 示例中的字段名要与正文规则一致。
- 自检项应覆盖数量、空间分布、非重叠、禁区、承载关系、修改边界和输出合规。
- 不要在最终输出要求中加入“思维链”“候选池”“内部推理过程”等内容；如需说明流程，应明确为内部执行且不要输出。

## 验证清单

提交修改前至少检查：

- 是否仍为 UTF-8 中文 Markdown。
- 是否保留用户提供内容边界，没有引导模型擅自补全。
- 是否保留灯光/发光/明暗相关禁令。
- 是否保留人物姿态、骨架、头身朝向限制。
- 若修改 `组图 23.md` 的人物规则，是否保留“同一人物面部表情区与衣领/领带/颈部、头发/头饰、耳饰/耳侧饰品互斥，且高于 Ref 复刻”的约束。
- 若修改 `组图 23.md` 的表情规则，是否允许 Ref 明确可见的嘴部承载物，同时禁止脑补不可见物、来源或动作过程。
- 是否保留大面积背景稳定限制。
- 是否保留数量配额和固定输出格式。
- 是否没有把“位置”字段写成外观描述字段。
- 是否没有引入与现有规则冲突的新术语。
